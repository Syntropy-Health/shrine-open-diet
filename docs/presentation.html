What is it?

A paid, Clerk-authenticated Model Context Protocol server that exposes a biochemical knowledge graph to AI agents.

Herbs → compounds → molecular targets → diseases, plus per-tenant clinical overlays.

Why MCP?

For the agent
  • No custom client SDK
  • Tool discovery via tools/list
  • Works in Claude Desktop, Cursor, custom agents
For the platform
  • Standard transport (Streamable HTTP)
  • Per-call authentication + metering
  • Scope-based tenant isolation

Mechanism

A thin adapter on top of a Python semantic engine.

The three-layer stack

Transport

TypeScript Streamable HTTP MCP server. Clerk JWT auth with kg:read / kg:write scopes. Per-call metering reports to SyntropyJournal.

Semantic Engine

Python LightRAG with a scope_filter that enforces tenant isolation server-side. Embedding pipeline via Ollama (test) or OpenAI (prod).

Graph Store

Neo4j with property-based scoping: scope: 'shared' or scope: 'tenant:X'. Traversal queries respect scope at the Cypher layer.

Tenant permission model

ScopeReadWrite
kg:read shared + own tenant:X —
kg:write — Own tenant:X only; shared is immutable

Tenants build on shared open-source knowledge without being able to corrupt it. Future: per-tenant visibility flag for opt-in cross-tenant sharing.

Entity types

  • Mechanism of Action

Clinical ontology (tenant-scoped)

Each clinic extends the shared graph with their own practice knowledge:

Protocol → Intervention → Outcome → Biomarker
          (clinic designs)   (observed)  (measured)

Example: "Adrenal Reset Protocol" includes Ashwagandha + Magnesium → cortisol:DHEA ratio improvement → measured via saliva panel. Written via tenant-ingest, visible only to that tenant.

How data flows in

  1. Ingestion pipeline: open-source data → LightRAG embeddings → Neo4j nodes/edges with scope: shared
  2. Tenant overlay: clinic POSTs structured entities via tenant-ingest MCP tool
  3. Query-time filter: scope_filter = ['shared', 'tenant:X'] injected into every Cypher query
  4. Audit log: per-tenant append-only log of every read and write

Applications

Who uses it, and how.

Application 1 — Longevity clinics

The practitioner

  • Naturopath, integrative MD, functional nutritionist
  • Uses Claude Desktop / Cursor as reasoning partner
  • Needs biochemistry grounded in evidence, not generic LLM guesses

The workflow

  • Ask "What foods support glutathione synthesis with elevated oxidative stress?"
  • Agent traverses shared KG: NAC → cysteine → glutathione pathway
  • Cross-references clinic's own Outcome data
  • Proposes protocol; practitioner approves

Application 2 — SyntropyJournal consumers

Self-hosters or hosted users who flip on the features.knowledge_graph feature flag.

  • ShrineAI's existing ReAct agent gets KG tools added to its tool list at startup
  • User asks: "What foods help my elevated CRP?"
  • Graphiti surfaces personal context; KG MCP surfaces shared evidence; combined response
  • Metered to user's subscription — consumer path to the same paid service

Application 3 — Partner / third-party agents

  • Other health-tech agents integrate via MCP discovery
  • Example: a Shopify protocol-builder agent queries KG to validate ingredient synergy before proposing a bundle
  • Example: a clinical trial screening agent queries KG for biomarker-disease-intervention links to match patients to protocols
  • Per-partner tokens, per-call metering, scope-filtered access

What makes the data defensible

Shared base

Open-source (licenses respected). Value isn't in the raw data — it's in the unified graph structure, the cross-DB linkages, and the embedding pipeline that enables semantic traversal.

Per-tenant overlay

Clinic's real-world protocol-outcome data is irreproducible from public sources. The overlay is where the compounding moat forms: more clinics → more tenant data → better benchmarks.

Connect

Claude Desktop config

{
  "mcpServers": {
    "syntropy-kg": {
      "url": "https://syntropy-kg-mcp-production.up.railway.app/mcp",
      "headers": {
        "Authorization": "Bearer <CLERK_JWT_WITH_kg:read>"
      }
    }
  }
}

Generate JWT in SyntropyJournal → Settings → API Tokens → select kg:read scope (requires active KG subscription).
